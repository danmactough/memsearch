memsearch plugin · shutdown bug

The infinite loop on exit.

A diagnosis of why Claude Code hangs when you close a session — and why every prior fix attempt was reverted.
repo: memsearch file: plugins/claude-code/ 5 reverted commits

Symptom

You type /exit. Claude Code never returns.

On session shutdown, the terminal hangs indefinitely. Activity Monitor shows a tree of claude / node / uvx memsearch processes spawning beneath the parent shell. Each child completes in ~10s, then spawns another, then another.

From the user's perspective: an infinite loop. From the OS's perspective: a deeply-nested chain of synchronous child invocations.

Git history shows five commits trying to fix this — all reverted: 49b8ac0, 56be020, 79251d9, ee7226e, 3d4dd1d. The current main still has the original, ineffective guard.

Architecture refresher

What the Stop hook actually does.

When a Claude Code session "stops" (model finishes responding), the memsearch Stop hook runs stop.sh, which summarizes the last turn into a daily Markdown file.

parent session └─ Stop hook fires └─ stop.sh ├─ parse-transcript.sh (extract last turn) ├─ claude -p --model haiku ... ← spawns a child Claude Code session ├─ append summary to YYYY-MM-DD.md └─ memsearch index $MEMORY_DIR

The child claude -p is what makes the summary. It's a fresh Claude Code session — its own transcript, its own hooks.

The misdiagnosis

The stop_hook_active guard is solving the wrong problem.

Current stop.sh tries to break the loop with this check:

STOP_HOOK_ACTIVE=$(_json_val "$INPUT" "stop_hook_active" "false")
if [ "$STOP_HOOK_ACTIVE" = "true" ]; then
  echo '{}'; exit 0
fi

But stop_hook_active is Claude Code's signal for an intra-session condition: "the Stop hook returned decision: block, Claude continued, and is now stopping again." It is set on the next Stop event in the same session.

A child claude -p is a brand-new session. Its first Stop event carries stop_hook_active=false. The guard never fires.

Root cause confidence 5 / 5

The child fires the same hook. Stop.sh reenters itself.

  1. User exits parent session → parent Stop fires → stop.sh runs.
  2. stop.sh runs claude -p --model haiku ... to summarize.
  3. Child Claude Code session boots, loads plugins (memsearch is user-scope, so it loads), runs the summarize prompt, then fires its own Stop.
  4. Child Stop → stop.sh runs in the child. stop_hook_active=false. Guard does not fire.
  5. Child stop.sh parses its transcript, finds the haiku turn, runs claude -p again. Grandchild.
  6. Each level takes ~10s. The shell waits on its child synchronously. Perceived as infinite.

The evidence

The env var marker is already there — just never checked.

Look at the claude -p invocation in current stop.sh:

SUMMARY=$(printf '%s' "$PARSED" | MEMSEARCH_NO_WATCH=1 CLAUDECODE= claude -p \
  --model haiku \
  --no-session-persistence \
  --no-chrome ...)

MEMSEARCH_NO_WATCH=1 is already being exported to the child env. Three helpers in common.sh consult it (start_watch, stop_watch, kill_orphaned_index), so child sessions skip watch management.

But the top of stop.sh never checks it. Of all places where the guard would have actually stopped recursion, the hook's own entry point is the one place it was forgotten.

Why the obvious fix doesn't work

--bare would break authentication.

Claude Code has a --bare flag that explicitly skips hooks. It was added in commit 49b8ac0 ("Update stop hook to avoid endless recursion"). Read its full description from claude --help:

--bare    Minimal mode: skip hooks, LSP, plugin sync, attribution,
          auto-memory, background prefetches, keychain reads, and
          CLAUDE.md auto-discovery. Sets CLAUDE_CODE_SIMPLE=1.
          Anthropic auth is strictly ANTHROPIC_API_KEY or apiKeyHelper
          via --settings (OAuth and keychain are never read).
This user is on a Claude subscription (OAuth). A --bare child can't authenticate → the summarizer silently produces nothing. --bare was the right intuition but the wrong knob.

The fix · part 1

Disable just memsearch in the child, via --settings.

--settings accepts inline JSON that merges on top of the resolved settings. We override enabledPlugins for memsearch only:

SUMMARY=$(printf '%s' "$PARSED" | MEMSEARCH_NO_WATCH=1 CLAUDECODE= claude -p \
  --model haiku \
  --settings '{"enabledPlugins":{"memsearch@memsearch-plugins":false}}' \
  --no-session-persistence \
  --no-chrome \
  --system-prompt "$SYSTEM_PROMPT")
Empirically verified by running the child with --include-hook-events --output-format stream-json --verbose: the init event lists only the other plugins. Memsearch is excluded; its hooks never fire.
  • Targets exactly the offending plugin — other plugins keep working.
  • Independent of where memsearch is installed (user / project / local).
  • Doesn't touch auth. OAuth keychain still works.

The fix · part 2 (defense in depth)

And actually check the env-var guard.

Promote MEMSEARCH_NO_WATCH=1 to a top-of-file guard in both stop.sh and session-start.sh. The variable is already exported on the claude -p line via VAR=val cmd — bash inherits it. We just need to read it at the entry point:

source "$SCRIPT_DIR/common.sh"

if [ "${MEMSEARCH_NO_WATCH:-}" = "1" ]; then
  echo '{}'
  exit 0
fi

And delete the dead stop_hook_active block while we're there — it was protecting against something that wasn't the problem.

Bonus: the session-start.sh guard also fixes Hypothesis 5 — child sessions stop racing the parent on the milvus_lite file lock.

Each attempt had the right idea but the wrong wiring.

With the --settings override plus the entry-point env-var guard, both layers of defense are in place: the child can't load the plugin in the first place, and even if a future regression undid that flag, the hook would still bail at line one.
